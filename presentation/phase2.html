⚡ Wo ist LLM beteiligt?

  • /exercises/submit → evaluate_exercise (1 LLM-Aufruf)
  • /exercises/hint → get_hint (1 LLM-Aufruf)
  • /skill-tests/generate → generate_skill_test (1 LLM-Aufruf)
  • /skill-tests/submit → evaluate_skill_test (2 LLM-Aufrufe)
  • /learning-progress/analyze → analyze_skill (1 LLM-Aufruf)

🛡️ Sicherheit — kein Client-Trust

  • Skill-Tests serverseitig gespeichert — Client schickt nur session_id
  • User kann Fragen nicht manipulieren
  • JWT Bearer Token bei jedem Request validiert
  • ExerciseCompletion is_locked = true → Wiederholung geblockt
Agent Tools — Sprint 2

5 neue LangChain Tools

⚖️
evaluate_exercise
agent/tools/exercise_evaluator_tool.py

Bewertet Schüler-Lösung in 3 Fällen: (1) stdout leer → falsch, (2) stdout == expected → LLM prüft ob Konzept stimmt oder hardcodiert, (3) stdout ≠ expected → LLM: teilweise oder falsch.

1 LLM-Aufruf
🏭
generate_exercise
agent/tools/exercise_generator_tool.py

Generiert dynamisch neue Übungsaufgaben für Intermediate & Advanced Skills. LLM erhält bereits abgeschlossene Titel — keine Wiederholung. Gibt JSON zurück: title, description, expected_output, hint.

1 LLM-Aufruf
💡
get_hint
agent/tools/hint_tool.py

Gibt gestuften Hinweis basierend auf hint_level (1–3). Level 1 = vage, Level 2 = konkreter, Level 3 = fast die Lösung. Lösung wird nie direkt verraten.

1 LLM-Aufruf
📋
generate_skill_test
agent/tools/skill_test_generator_tool.py

Generiert vollständigen Test: 3 Multiple-Choice + 1 Code-Lesen + 1 Mini-Aufgabe. Test wird serverseitig gespeichert — Client bekommt nur session_id, kann Fragen nicht manipulieren.

1 LLM-Aufruf
🎓
evaluate_skill_test
agent/tools/skill_test_evaluator_tool.py

Bewertet Test in 3 Teilen: MC = reiner String-Vergleich (kein LLM, 30 Punkte), Code-Lesen = LLM semantisch (30 Punkte), Mini-Aufgabe = Code ausführen + LLM bewertet (40 Punkte). Bestanden ≥ 60/100.

2 LLM-Aufrufe
💬
Warum direkte invoke() Aufrufe?

Diese 5 Tools werden nicht über den ReAct-Agent-Loop aufgerufen — sondern direkt via tool.invoke() aus dem Router. Grund: deterministisches Verhalten, kein LLM-Entscheidungsschritt nötig.

Kein Agent-Loop
💡

Tool-Docstring = LLM-Beschreibung: Jedes @tool-Docstring ist die Beschreibung die das LLM sieht wenn es entscheiden muss welches Tool zu rufen. Guter Docstring = korrektes Tool.
