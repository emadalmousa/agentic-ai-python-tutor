🤖 KI-Agent — agent/tutor_agent.py  ·  ⛓ LangChain create_agent (LangGraph)
Was ist ein KI-Agent — und was ist der Unterschied zu einem Chat-Bot?
Ein normaler Chat-Bot bekommt eine Frage und antwortet direkt. Ein Agent bekommt eine Aufgabe und entscheidet selbst welche Werkzeuge (Tools) er aufruft, in welcher Reihenfolge, und ob er das Ergebnis eines Tools als Input für das nächste nutzt. Er hat Werkzeuge zur Verfügung — und Verstand genug um sie richtig einzusetzen.
🔄 Der ReAct-Loop — Reason → Act → Observe agent/tutor_agent.py → run_analysis()
🧠
1. REASON
Das LLM liest den Code und überlegt: „Was brauche ich zuerst?" → entscheidet: explain_code_tool aufrufen
⚡
2. ACT
LangGraph führt das gewählte Tool aus. Das Tool ruft intern das LLM erneut auf und gibt ein Ergebnis zurück.
👁
3. OBSERVE
Das Agent-LLM sieht das Tool-Ergebnis und entscheidet: noch ein Tool? Oder ist alles da für die Final Answer?
Dieser Loop wiederholt sich bis der Agent alle nötigen Informationen hat → dann schreibt er die Final Answer im vorgegebenen Format.
⛓ Wie LangChain hier konkret genutzt wird agent/tutor_agent.py
# LangChain-Import: create_agent kommt aus langchain.agents (LangChain 1.3 / LangGraph)
from langchain.agents import create_agent

# 1. LLM holen (OpenAI oder Ollama — entschieden in config.py)
llm = get_llm() # → ChatOpenAI oder ChatOllama

# 2. Tool-Liste bauen — rag_tool nur wenn FAISS-Index vorhanden
tools = _build_tools() # [explain_tool, debug_tool, exercise_tool, (rag_tool)]

# 3. LangChain erstellt einen LangGraph-Agenten (CompiledStateGraph)
agent = create_agent(llm, tools, system_prompt=_SYSTEM_PROMPT)

# 4. Agent wird mit der Code-Aufgabe gestartet → ReAct-Loop läuft
result = agent.invoke({"messages": [("human", f"Analysiere: {code}")]})

# 5. Letzte Nachricht = Final Answer des Agents
final_text = result["messages"][-1].content

# 6. Regex-Parser extrahiert 5 Felder aus dem Free-Text
parsed = _parse_agent_output(final_text)
create_agent aus langchain.agents Agent entscheidet selbst welche Tools
_build_tools() — warum rag_tool bedingt ist:
_build_tools() prüft ob der FAISS-Ordner (backend/vectorstore/) existiert. Nur wenn ein PDF hochgeladen wurde existiert dieser Ordner. Ohne Index würde rag_tool beim Aufruf abstürzen. Der Agent bekommt also nur die Tools die er tatsächlich nutzen kann.
_parse_agent_output() — warum der Parser nötig ist:
Der ReAct-Agent gibt seine Final Answer als freien Text aus. _parse_agent_output(text) sucht per Regex nach Labels (Erklärung:, Fehler gefunden:, etc.) und extrahiert daraus das strukturierte Dict. Fehlt ein Feld → sicherer Default. Dieser Parser hält das Router-Interface stabil egal wie der Agent formuliert.
LangChain-Rolle hier: create_agent() aus LangChain 1.3 erzeugt einen LangGraph CompiledStateGraph. LangGraph managt den Reason→Act→Observe-Loop, verwaltet den Nachrichten-State und ruft bei jedem „Act"-Schritt das passende Tool auf — alles automatisch, ohne dass man den Loop selbst schreiben muss.
⛓ LangChain Tools  ·  agent/tools/  ·  jedes Tool = ein LLM-Aufruf
⛓Was LangChain bei den Tools macht — und was es NICHT ist
LangChain ist kein KI-Modell selbst — es ist das Framework das:
Was LangChain tut:
• @tool-Decorator: macht eine Python-Funktion zum Agent-Werkzeug
• SystemMessage / HumanMessage: strukturiert den Prompt
• llm.invoke([...]): sendet an OpenAI/Ollama, empfängt Antwort
• Abstrahiert HTTP-API — Tool-Code ist provider-unabhängig
Was die KI tut:
Das LLM (gpt-4o oder llama3.2) liest den System-Prompt, versteht den Code und generiert die Antwort Token für Token. LangChain schickt nur die Anfrage und empfängt das Ergebnis — die Intelligenz kommt vom Modell selbst.

🧠Was das Modell wirklich macht
Hier passiert die eigentliche Intelligenz.

Das Modell wurde auf Milliarden Code-Beispielen trainiert. Es:
• erkennt Python-Muster und Fehler durch dieses Training

📚 RAG-Pipeline  ·  agent/rag/  ·  ⛓ LangChain Embeddings
Was ist RAG und warum?
Ohne RAG kennt der Agent nur was das LLM im Training gesehen hat — er kann keine spezifischen Kursinhalte kennen. Mit RAG kann ein Lehrer sein Kursbuch als PDF hochladen. Bei jeder Analyse sucht der Agent nach passenden Stellen und kann daraus zitieren. Die Antworten sind dadurch direkt auf das Lernmaterial des Kurses abgestimmt.
1
loader.py — PDF → Text
extract_text(pdf_bytes) — pypdf.PdfReader liest alle Seiten. Gibt einen einzigen Text-String zurück. Wirft ValueError wenn PDF leer. 🚫 kein LangChain
2
splitter.py — Text → Chunks
split_text(text) — rekursives Aufteilen: zuerst an \n\n, dann \n, dann Leerzeichen. Chunk-Größe 500 Zeichen, 50 Zeichen Überlapp. Überlapp verhindert Informationsverlust an Grenzen. 🚫 kein LangChain
3
vectorstore.py — Chunks → Vektoren → FAISS-Index
build_and_save(chunks):
get_embeddings().embed_documents(chunks) — hier LangChain! Wandelt Text in Zahlenvektoren um (via OpenAI oder Ollama).
faiss.IndexFlatL2(dim) → Index aufbauen → als index.faiss + chunks.pkl speichern. get_embeddings() = LangChain FAISS IndexFlatL2
4
Antwort → Frontend
{"status": "ok", "chunks": 142} — ab sofort nutzt jede Analyse den FAISS-Index für Quellen.
Wie funktioniert Vektorsuche (Embeddings)?
Jeder Text-Chunk wird in einen Zahlenvektor mit ~1500 Dimensionen umgewandelt (z.B. [0.23, -0.81, 0.04, ...]). Ähnliche Texte haben ähnliche Vektoren. Bei einer Suchanfrage wird die Anfrage ebenfalls zu einem Vektor — FAISS findet dann die Chunks mit der kleinsten Distanz zu diesem Vektor. Das ist semantische Suche: nicht Keyword-Matching, sondern Bedeutungsähnlichkeit.

🔑 Wo LangChain und der Agent eine Rolle spielen — Zusammenfassung
Jede Schicht — wer macht was
⛓ LangChain — @tool Decorator
Macht Python-Funktionen zu Agent-Werkzeugen. Der Docstring wird die Beschreibung die der Agent liest wenn er überlegt welches Tool er aufruft. Ohne @tool kennt der Agent das Werkzeug nicht.
⛓ LangChain — create_agent()
Erzeugt einen LangGraph CompiledStateGraph. Verwaltet den Reason→Act→Observe-Loop automatisch. Man gibt LLM + Tools → LangChain baut den Agenten-Loop.
⛓ LangChain — SystemMessage / HumanMessage
Strukturieren den Prompt für das LLM. SystemMessage = Instruktion, HumanMessage = Eingabe. Genutzt in allen 4 Tools und im Chat-Endpunkt direkt.
⛓ LangChain — ChatOpenAI / ChatOllama
Provider-Abstraktion. Beide Klassen haben llm.invoke([...]) — identische Schnittstelle. Wechsel des Providers = eine Zeile in config.py.
⛓ LangChain — OpenAIEmbeddings / OllamaEmbeddings
Wandeln Text in Zahlenvektoren um. Genutzt in vectorstore.py für RAG. Gleiche Provider-Logik wie ChatOpenAI/ChatOllama.
🤖 KI-Agent — Reasoning
Der Agent liest den System-Prompt + Code-Aufgabe und entscheidet selbst welche Tools er aufruft. Ein Chat-Bot würde das nicht tun — er antwortet direkt.
🤖 KI-Agent — Tool-Auswahl
Der Agent kann rag_tool überspringen wenn kein Lernmaterial hochgeladen wurde, Tools in anderer Reihenfolge aufrufen, oder ein Tool mehrfach aufrufen. Das ist adaptive Intelligenz — kein festes Skript.
📚 RAG — Vektorsuche
Kein Agent-Feature, kein LangChain-Feature — sondern eine separate Pipeline. PDF → Chunks → Embeddings → FAISS-Index. Der Agent nutzt rag_tool als einen seiner möglichen Werkzeuge.
🐍 FastAPI + Pydantic
HTTP-Routing, Request-Validierung, Response-Serialisierung. Kein KI-Wissen. Trennt sauber HTTP-Schicht von Agent-Logik.
🚫 subprocess — kein LLM
POST /tutor/run ruft den Python-Interpreter direkt auf. Kein LangChain, kein Agent, kein LLM. Schnell, kostenlos, zeigt echte Python-Fehler.
⚙ config.py — Provider-Gate
Einzige Stelle wo OpenAI vs. Ollama entschieden wird. Alle anderen Module rufen nur get_llm() auf — kein Provider-Wissen außerhalb von config.py.
🖥 Next.js Frontend
UI-Rendering, State, HTTP-Requests. Kein KI-Wissen. Empfängt fertige JSON-Antworten und zeigt sie gerendert an (Markdown, Syntax-Highlighting).
